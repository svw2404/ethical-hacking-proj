// Blue Team — Logging & Monitoring Explanation

OWASP Category: A09:2021 — Security Logging and Monitoring Failures

This log demonstrates what a real application should record: who attempted to log in, from which IP address, whether the payload looked malicious, whether access was unauthorized, and what the outcome was.

Why logging matters:

  • Without logs, repeated attack attempts go unnoticed until serious damage occurs.
  • Logs allow incident response teams to reconstruct exactly what happened and when.
  • Alerts can be triggered when suspicious patterns appear (e.g., 100 failed logins from one IP).
  • Logs must be tamper-resistant and stored separately from the application they monitor.
  • Logs should never contain raw passwords or full credit card numbers.

Hardening plan:

  • Send logs to a centralized SIEM (Security Information and Event Management) system.
  • Set up alerting rules for high-frequency login failures or SQL injection signatures.
  • Store logs with write-only access — applications can append but not delete.
  • Rotate and archive logs regularly with integrity verification (e.g., hash chains).
