// Blue Team Explanation

Logging and monitoring are essential for detecting suspicious activity, investigating incidents, and proving what happened during an attack attempt. A system may still be vulnerable, but without logs the defenders may not know that the attack occurred.

OWASP Category A09:2021 — Security Logging and Monitoring Failures
Purpose Record authentication attempts, suspicious input, blocked access, and system errors.
Blue Team Use Review evidence after Red Team activity and explain detection strategy.

// Events This Lab Records

Event Type Level Purpose
Login attempt INFO Tracks authentication activity by username and source IP.
Suspicious input pattern WARNING Marks input that resembles an injection or abuse attempt.
Login failure WARNING Helps identify brute-force or repeated failed access attempts.
Login success INFO Records successful authentication events.
Unauthorized profile request WARNING Detects possible broken access control or IDOR behavior.
Rate limit event WARNING Shows repeated authentication attempts from the same source.
System/database error ERROR Helps defenders detect unexpected failures or attack side effects.

// Why Logging Matters

  • Logs help defenders notice attacks that would otherwise remain invisible.
  • Logs allow teams to reconstruct the timeline of an incident.
  • Logs support alerting rules for repeated failures or suspicious input patterns.
  • Logs should be protected from tampering and stored separately in real systems.
  • Logs should avoid recording raw passwords, full payment data, or unnecessary sensitive information.

// Long-Term Hardening Plan

  • Send logs to a centralized SIEM system.
  • Create alerts for repeated failed logins and suspicious request patterns.
  • Use write-only application log permissions where possible.
  • Rotate, archive, and integrity-check logs regularly.
  • Review logs as part of incident response and security monitoring.
